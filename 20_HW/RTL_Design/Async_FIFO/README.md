# Async FIFO

서로 다른 클럭 도메인(Read/Write) 간에 데이터를 안전하게 전달하고 메타스태빌리티를 방지하는 비동기 FIFO 설계 및 자동화 검증.

---

## 개념

### 왜 필요한가

- 생산자와 소비자가 서로 다른 클럭 도메인을 사용할 때, 신호를 바로 넘기면 메타스태빌리티가 발생할 위험이 있다.
- 이를 해결하기 위해 쓰기와 읽기가 각자의 클럭으로 독립 동작하며 데이터를 안전하게 전달하는 Async FIFO가 필요하다.

### 내부 구조

| 요소                      | 역할                                            |
| ----------------------- | --------------------------------------------- |
| 메모리 배열                  | 실제 데이터를 저장하는 레지스터 (depth x width, wr_clk 도메인) |
| wr_ptr / rd_ptr         | 각 클럭 도메인의 바이너리 포인터                            |
| 그레이 코드 변환기              | 도메인 경계를 안전하게 넘기기 위한 binary → gray 변환기         |
| 2-stage FF Synchronizer | 다른 클럭 도메인의 그레이 포인터를 동기화하여 메타스태빌리티 해소          |

### 왜 그레이 코드를 쓰는가?

- 바이너리 포인터는 상태가 변할 때 여러 비트가 동시에 변하므로, 클럭 도메인 경계에서 샘플링 시 중간에 깨진 값이 보일 수 있다.
- 그레이 코드는 인접한 값 사이에 **1비트만** 변하기 때문에, 샘플링 타이밍이 어긋나더라도 이전 값이나 다음 값만 보이게 되어 안전하다.

### **Full / Empty 판별:** Gray 코드 비교

- 포인터는 Sync FIFO와 동일하게 Extra bit 방식을 유지하며, 비교는 동기화된 그레이 코드 포인터로 수행한다.

| 상태 | 판단 도메인 | 조건 |
|------|------------|------|
| Empty | rd_clk | `wr_ptr_gray_sync == rd_ptr_gray` (전체 비트 일치) |
| Full | wr_clk | 상위 2비트 반전, 나머지 비트 일치 |

-  **보수적(Pessimistic) 판단 로직:**
	- 2-stage 동기화를 거친 포인터는 실제 상태보다 약간 과거의 값이다. 
	- 따라서 실제보다 Empty(읽기 제한)와 Full(쓰기 제한) 상태를 약간 더 길게 유지하게 되는데, 이는 데이터 유실이나 오버라이트를 완벽하게 차단하므로 시스템 안정성 측면에서 안전(Safe)하다.

## 설계 결정

**바이너리 포인터 별도 유지**
- 메모리 주소 접근은 바이너리 포인터로 수행하고, 클럭 도메인 경계를 넘길 때만 그레이 코드로 변환한다. 
- 그레이 코드는 순차적 증가가 보장되지 않아 주소 인덱스로 직접 쓸 수 없기 때문이다.

**Combinational Read (assign rd_data)**
- Registered Read 사용 시 발생하는 1사이클 레이턴시를 없애기 위해 `assign`을 사용했다.
- 이로 인해 `rd_ptr` 기준으로 즉시 데이터가 출력되어 TB 타이밍 불일치 문제를 깔끔하게 해결했다.

**C 모델의 WAIT 로직 삽입**
- C 모델에는 클럭 지연(Sync latency) 개념이 없기 때문에 RTL과 동작 타이밍에 차이가 발생한다. 
- RTL이 Full 상태에서 회복되는 시간을 벌어주기 위해, 
- C 모델에서 prev_full && !cur_full 조건을 감지하면 stimulus에 `WAIT` 커맨드를 삽입하도록 하여 완벽한 교차 검증 환경을 구축했다.

## 검증 결과 (검증 파이프라인 자동화)

Sync FIFO에서 개선 예정이었던 **C 모델 랜덤 입력 자동 생성**, **TB 자동 구동**, **결과 자동 비교** 환경을 Makefile 기반으로 100% 구축 완료했다.

- **stimulus.txt:** C 레퍼런스 모델이 생성한 랜덤 입력 패턴
- **golden_output.txt:** C 레퍼런스 모델이 예측한 정답 데이터
- dut_output.txt: RTL 시뮬레이션 결과 (타임스탬프 @ 포함)

`compare.exe`를 통한 자동 비교 결과, READ 데이터의 무결성과 순서가 C 모델과 완전 일치(PASS)함을 확인했다.

```text
PASS: @ 75000
  Golden READ : data=42
  DUT    READ : data=42
PASS: @ 105000
  Golden READ : data=17
  DUT    READ : data=17
...
Total PASS:    20
Total FAIL:    0
Total SKIPPED: 2
All 20 cases processed.
